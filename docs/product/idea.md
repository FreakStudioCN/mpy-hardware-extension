Package Intelligence 层聚合 uPyPI、GraftSense 和 curated recipes 中的 MicroPython 包/驱动，覆盖温湿度、气压、IMU、显示屏、音频编解码等主流器件。它提供产品级 API：
- GET /v1/packages/search?q=温度 → 返回相关包候选、能力标签和置信等级。
- POST /v1/packages/resolve → 根据意图、能力和开发板约束选择候选包。
- GET /v1/packages/context/{name} → 返回 driver context（作者、版本、依赖、import、构造函数、读写方法、总线、已知问题）。
AI Skill 通过 driver context 自动使用最合适的真实驱动，而不是“幻觉”出一个不存在的函数名。

生成代码示例（简化）：
用户一句话：“做一个环境监测器，显示温湿度、光照，并在温度超过 30 度时让马达震动。”
Skill 生成的 main.py 框架：
```Python
from upyos import Activity
import lvgl as lv
from drivers import AHT20, BH1750, VibrationMotor  # 自动从 driver context 匹配
import asyncio

class EnvMonitorActivity(Activity):
    def on_create(self, bundle):
        # 初始化传感器
        self.i2c = I2C(0, scl=Pin(22), sda=Pin(21))
        self.aht = AHT20(self.i2c)
        self.bh = BH1750(self.i2c)
        self.motor = VibrationMotor(Pin(15))
        
        # 创建 UI
        self.label = lv.label(lv.scr_act())
        self.label.set_text("温湿度/光照")
        
    def on_resume(self):
        self.timer = lv.timer_create(self.update, 2000, None)  # 2 秒刷新
        
    def update(self, timer):
        temp, hum = self.aht.read()
        lux = self.bh.read()
        self.label.set_text(f"温度:{temp:.1f}°C  湿度:{hum:.0f}%\n光照:{lux:.0f}lx")
        if temp > 30:
            self.motor.vibrate(0.5)  # 震动 0.5 秒```
用户只需将生成的代码放到 uPyOS 设备的 /apps/ 目录，重启即可运行。

2.5 终端侧模块管理与应用分发
用户上电主动式模块（自动蓝牙广播），终端扫描识别后，自动匹配对应应用，每个模块类型对应一个推荐的 系统应用 或 第三方应用。应用开发者可以在 manifest.json 中声明支持的模块类型：
```JSON
{
  "app_id": "com.envmonitor",
  "supported_modules": [
    { "type": 0x01, "min_ver": "1.0" },
    { "type": 0x02 }
  ]
}```
终端发现模块时，会优先启动匹配的应用，并将模块的数据流自动注入到应用的 on_module_data() 回调中。
对于传统的有线模块（I2C、UART、GPIO），终端依然支持通过外接 Grove 接口（或排针）连接。AI 生成的代码中，对于这些模块需要明确引脚定义。但我们的 最终愿景 是鼓励开发者将复杂的传感器封装成主动模块，从而让 AI 只需处理极简的统一接口。

3.1 去编译化：AI 生成 Python 的天然优势
传统嵌入式开发中，哪怕改一行代码，也需要完整的编译-烧录-运行循环，一次迭代至少 30 秒。而 MicroPython 是解释型语言：
- 无需编译：生成 .py 文件直接复制到设备，或通过 REPL 动态输入，即刻看到效果。
- REPL 调试：AI 可以一边生成代码，一边向设备发送指令，实时修正错误，完全自动化调优。
- 热更新：应用安装只是文件拷贝，卸载只是删除文件，不需要重新烧录整个固件。
类比：C SDK 的开发模式像刻印活字——改一个字要重铸整个版；MicroPython 像铅笔写字——涂改、重写任意自然。
这一点对于 AI 生成代码至关重要：因为大模型不能保证一次生成完全正确，必须支持快速迭代反馈回路。Python + REPL 是唯一能在 MCU 上提供毫秒级反馈的路径。
3.2 动态应用安装：C SDK 永远做不到的“应用商店”
传统基于 C 的嵌入式 SDK（如 ESP-IDF、Arduino 框架）编译时将所有的应用逻辑与驱动静态链接进固件。要增加一个新应用，必须重新编译整个固件并烧录，这实际上杀死了应用商店的可能。
我们的 uPyOS 实现了真正的动态加载：
- 应用是独立的 .py 或 .mpy 文件，存放在文件系统 /apps/ 目录。
- 系统启动器扫描该目录，动态导入并显示图标。
- 应用可以独立更新、删除，互不影响。
- 依赖的驱动包由 Package Intelligence 解析后按需下载，自动安装。
类比：C SDK 的设备就像功能手机，出厂时功能就焊死了；uPyOS 设备就像智能手机，随时从应用商店下载新 App。
3.3 主动模块：让 AI 只需关心“业务逻辑”，而非“寄存器”
对于多传感器组合的场景，如果每个模块的通信协议都不同，AI 生成的代码会极其臃肿且易错。我们的主动模块将所有传感器封装成独立的小主机（ESP32-C3 + 传感器），对外只暴露统一的数据接口：
- 上电后通过蓝牙广播：模块类型、数据格式、上报周期。
- 与终端之间采用简单 JSON 帧通信（通过 BLE GATT 或 ESP‑NOW）。
- 终端上的通用模块管理器自动解析、分流数据到对应的应用。
这样一来，AI 生成应用时完全不需要知道模块内部是什么传感器、如何初始化、如何读取。它只需要调用 get_module_data() 拿到温湿度、气体浓度等标准化数值，然后专注于业务逻辑（例如绘制图表、比较阈值、发送告警）。
类比：传统方式像要自己造螺丝、扳手、发动机才能组装一辆车；主动模块像 Lego 积木——每个模块自带接口说明，你只管拼搭创意。
