## 2. Technical Core Architecture

### 2.1 Overall Architecture

[Image]

uPyOS, built on MicroPython and LVGL, achieves the following on just a few hundred KB of RAM:

- **Activity lifecycle**: `on_create` / `on_resume` / `on_pause` / `on_destroy`, letting developers manage page state like writing an Android App.
- **System services**: WiFi configuration, OTA updates, AppStore client, module scanning.
- **Cross-MCU compatibility**: The same `.py` application can run on ESP32, ESP32-S3, RP2040, and other chips.

**Abstract analogy**: If embedded development is driving, the traditional way means you need to know how to repair engines, change tires, and tune suspensions before you can hit the road; uPyOS gives you an automatic car with a steering wheel and gas pedal — you only need to care about where you're going.

The Package Intelligence layer aggregates MicroPython packages and drivers from uPyPI, GraftSense, and curated recipes, covering mainstream components such as temperature/humidity, barometric pressure, IMU, displays, and audio codecs. It provides product-level APIs:

- `GET /v1/packages/search?q=temperature` → returns package candidates, capability tags, and confidence levels.
- `POST /v1/packages/resolve` → selects package candidates from intent, capabilities, and board constraints.
- `GET /v1/packages/context/{name}` → returns driver context: author, version, dependencies, imports, constructors, read/write methods, bus, and known issues.

The AI Skill uses driver context to select and call real drivers, rather than "hallucinating" a function name that doesn't exist.

**Abstract analogy**: In traditional development, finding a driver is like digging through a flea market; Package Intelligence is like a standardized e-commerce platform — search to find, one click to "purchase" (install).

**Generated code example (simplified):**

User's one sentence: "Make an environment monitor that shows temperature, humidity, and light, and makes the motor vibrate when the temperature exceeds 30 °C."

The `main.py` skeleton generated by the Skill:

```python
from upyos import Activity
import lvgl as lv
from drivers import AHT20, BH1750, VibrationMotor  # auto-matched from driver context
import asyncio

class EnvMonitorActivity(Activity):
    def on_create(self, bundle):
        # init sensors
        self.i2c = I2C(0, scl=Pin(22), sda=Pin(21))
        self.aht = AHT20(self.i2c)
        self.bh = BH1750(self.i2c)
        self.motor = VibrationMotor(Pin(15))

        # build UI
        self.label = lv.label(lv.scr_act())
        self.label.set_text("Temp/Humidity/Light")

    def on_resume(self):
        self.timer = lv.timer_create(self.update, 2000, None)  # refresh every 2s

    def update(self, timer):
        temp, hum = self.aht.read()
        lux = self.bh.read()
        self.label.set_text(f"Temp:{temp:.1f}°C  Hum:{hum:.0f}%\nLight:{lux:.0f}lx")
        if temp > 30:
            self.motor.vibrate(0.5)  # vibrate for 0.5s
```

The user simply drops the generated code into the uPyOS device's `/apps/` directory and reboots to run it.

### 2.5 Terminal-Side Module Management and App Distribution

When the user powers on an active module (which automatically Bluetooth-broadcasts), the terminal recognizes it after scanning and automatically matches the corresponding app. Each module type maps to a recommended **system app** or **third-party app**. App developers can declare supported module types in `manifest.json`:

```json
{
  "app_id": "com.envmonitor",
  "supported_modules": [
    { "type": 0x01, "min_ver": "1.0" },
    { "type": 0x02 }
  ]
}
```

When the terminal discovers a module, it prioritizes launching the matched app and automatically injects the module's data stream into the app's `on_module_data()` callback.

For traditional wired modules (I2C, UART, GPIO), the terminal still supports connection via external Grove ports (or pin headers). AI-generated code for these modules must declare explicit pin definitions. But our **ultimate vision** is to encourage developers to encapsulate complex sensors as active modules, so the AI only has to deal with a minimal unified interface.

### 3.1 De-compilation: The Natural Advantage of AI Generating Python

In traditional embedded development, changing even one line of code requires a full compile-flash-run cycle — at least 30 seconds per iteration. MicroPython, however, is an interpreted language:

- **No compile**: Generated `.py` files can be copied directly to the device or typed dynamically via REPL, giving instant results.
- **REPL debugging**: The AI can generate code while sending commands to the device, fixing errors in real time — fully automatable tuning.
- **Hot update**: App installation is just a file copy; uninstall is just a file delete. No firmware reflash needed.

**Analogy**: The C SDK development model is like movable-type printing — changing one character means recasting the whole plate; MicroPython is like writing with a pencil — erase and rewrite at will.

This matters enormously for AI code generation, because large models cannot guarantee a perfect first generation — they need a fast iterative feedback loop. Python + REPL is the only path that delivers millisecond-level feedback on an MCU.

### 3.2 Dynamic App Installation: An "App Store" the C SDK Can Never Have

Traditional C-based embedded SDKs (such as ESP-IDF or the Arduino framework) statically link all application logic and drivers into the firmware at compile time. Adding a new app requires recompiling and reflashing the entire firmware — which kills any chance of an app store.

Our uPyOS achieves true dynamic loading:

- Apps are independent `.py` or `.mpy` files stored in the `/apps/` directory.
- The system launcher scans the directory, dynamically imports, and shows icons.
- Apps can be updated or deleted independently, without affecting each other.
- Required driver packages are resolved by Package Intelligence, downloaded on demand, and installed automatically.

**Analogy**: A C-SDK device is like a feature phone — its functions are welded in at the factory; a uPyOS device is like a smartphone — you can download new apps from the store anytime.

### 3.3 Active Modules: Let the AI Worry About "Business Logic," Not "Registers"

For multi-sensor scenarios, if each module has a different communication protocol, the AI-generated code becomes bloated and error-prone. Our active modules wrap each sensor into a tiny independent host (ESP32-C3 + sensor) that exposes only a unified data interface to the outside world:

- After power-up it Bluetooth-broadcasts its module type, data format, and report period.
- Communicates with the terminal via simple JSON frames (over BLE GATT or ESP-NOW).
- The terminal's generic module manager parses and routes data automatically to the matching apps.

This way, when the AI generates an app, it doesn't need to know what's inside the module, how to initialize it, or how to read it. It only calls `get_module_data()` to get standardized values (temperature/humidity, gas concentration, etc.) and focuses on business logic (drawing charts, comparing thresholds, sending alerts).

**Analogy**: The traditional way is like having to forge your own screws, wrenches, and engine before you can assemble a car; active modules are Lego bricks — each one comes with its own interface spec, you just snap them together.
