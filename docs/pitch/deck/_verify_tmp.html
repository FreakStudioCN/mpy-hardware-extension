// the pain

Makers don't lack ideas 鈥?they get stuck on the path to running hardware.

  • Selection is hard: ESP32 / Arduino / STM32, sensors, actuators, comms 鈥?too many choices.
  • Wiring is hard: I2C / SPI / UART / ADC, voltage, pin muxing 鈥?one wrong step and nothing runs.
  • Drivers are hard: docs scattered across GitHub, vendor PDFs, forums, and sample code 鈥?AI happily hallucinates APIs that don't exist.
  • Debugging is hard: hardware errors aren't normal compile errors 鈥?serial logs, power, wiring, and firmware versions all tangled together.

Today's barrier isn't imagination. It's getting the thing to actually run.

~/blockless/deck/06_data-flow.md 06 / 14

// the real data flow behind one sentence

"Build hardware from a sentence" isn't a chat box 鈥?it's a shippable pipeline.

  1. User states the need: a temp/humidity monitor that lights up and shows the value above a threshold.
  2. System gets a hardware picture: bind the device, run an I2C scan, check off modules, or describe the hardware in plain language.
  3. The agent matches capabilities: it figures out the needed hardware_tags 鈥?display, sensing, comms, actuators.
  4. It links Package Intelligence: package.json, READMEs, driver context, deps, chip support, firmware requirements.
  5. It generates a MicroPython project: code, drivers, assets, and config files together.
  6. It packages to .mpk and writes MANIFEST.JSON 鈥?downloadable, pushable, editable online, or published to the store.
  7. The local IDE / VS Code extension calls mpremote 鈥?installs deps, uploads, runs, reads serial, auto-repairs.

The sentence is just the entrance. The moat is behind it: hardware profiling, driver retrieval, app packaging, local execution, and feedback-driven repair.

~/blockless/deck/09_package-intelligence.md 08 / 14

// package intelligence / ide

"Finding packages is hard and messy." We put package management in the workflow.

  • uPyPI + GraftSense + curated recipes index 200+ MicroPython packages/drivers 鈥?sensors, comms modules, and human-machine interaction.
  • Package details carry machine-readable info: name, version, description, author, license, chip support, firmware deps, file lists.
  • The Thonny plugin supports in-IDE search, metadata view, and download caching, and installs to the board's /lib via mpremote.
  • It handles multi-file packages, dependencies, network failures, unconfigured mpremote, and disconnected boards.

Our AI doesn't invent drivers 鈥?it calls a structured, installable, verifiable MicroPython package ecosystem.
